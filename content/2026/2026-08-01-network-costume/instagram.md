You don't have a network problem. You have an operational-discipline problem wearing a network costume. 🎭

The network is where problems become visible — so it's where blame lands. Packets drop where they drop, not where the decision that dropped them was made.

What gets called a "network problem":
▪️ The outage after a change window — caused by a change nobody reviewed.
▪️ The "flaky WAN" — caused by a config nobody has touched in three years.
▪️ The "multicast problem" I chased for days — the network was fine; it was exposing an application design flaw.

Same costume, different bodies: changes without review, fixes without documentation, intent living in one engineer's head.

The test: if the fix is new hardware, what happens when the same team with the same process runs the new platform? Same problems, newer fonts. Discipline doesn't ship in a box.

Fix the discipline and the network gets boring. Skip it — you've dressed the same problem in a newer costume.

What was your last "network problem" that was really a process problem? 👇
