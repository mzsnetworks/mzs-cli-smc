A runbook nobody runs isn't documentation. It's fiction with a table of contents.

We write the failover procedure, file it in the wiki, and feel safer. The safety is imaginary.

A runbook is only true the day it's written. The network keeps moving — the jump host in step 4 gets decommissioned, the CLI changes in a firmware upgrade — and the doc has no idea. It goes stale wearing the authority of something once correct.

You find out at 3am, when step 4 points to a server gone a year ago. Now you're debugging the outage and the runbook at once.

The value was never in writing it — it's in running it. Automate it so it fails loudly when it drifts, or rehearse it on a quiet Tuesday. The only runbook you can trust is one that gets exercised.
