Most internal automation isn't abandoned. It was never adopted.

The story is always the same. Someone builds a genuinely clever tool — a provisioning script, a config auditor, a self-service portal. It works. They demo it. Everyone nods. And six months later it's three commits stale and one person still runs it by hand.

The code shipped. The outcome didn't.

That gap is product thinking, and network teams almost never apply it to their own tools. We treat internal automation as a coding problem — write it, it works, done. But a tool that works and a tool that gets used are different achievements. The second one is a product problem.

Product thinking is just a few unglamorous questions the builder skipped:

— Who is the user, and what job are they hiring this to do?
— Is it easier than the manual way they already trust? If not, they won't switch.
— Can someone else run it without the author in the room?
— Who maintains it when the author changes teams?
— How does a new engineer even discover it exists?

Skip those and you don't have a platform. You have a personal script with good intentions, and its blast radius is one person's memory.

This is why "we automated that" so often means "one engineer automated that, for themselves." The automation exists. The capability didn't spread. The org still absorbs the manual toil everywhere that engineer isn't.

Working code is table stakes. Adoption is the actual deliverable — and adoption is a product outcome, not a technical one. If nobody but the author uses it, the network team shipped effort, not leverage.

Build the tool like it has users. Because the ones worth building do.
